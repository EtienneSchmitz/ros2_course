Le projet final met bout à bout tout ce que vous avez construit pendant la semaine :
un **système autonome complet** qui détecte, saisit et transporte un objet.

<Aside type="note" title="Le scénario">
La **caméra** détecte un objet et sa classe (vision, Jour 4) → le **SO-101** le saisit
(manipulation, Jour 3) → le **LeKiwi** le transporte vers une **zone de dépôt liée à sa
classe** (navigation, Jour 2) → dépôt. Un pipeline ROS 2 complet :
**perception → décision → action**.
</Aside>

## Les trois sous-systèmes

<Steps>

1. **Perception** — la caméra identifie un **objet** et sa **classe** (couleur, label,
   forme…). Une IA de vision (Jour 4) distingue **3 à 6 classes**. Sortie : la classe + la
   pose de l'objet.

2. **Manipulation** — le **SO-101** (Jour 3) saisit l'objet à la pose détectée et le
   dépose sur le robot mobile (ou une position connue).

3. **Navigation** — le **LeKiwi** (Jour 2) se déplace jusqu'à l'objet, le prend en charge,
   puis le transporte jusqu'au **point de dépôt associé à la classe** détectée.

</Steps>

## Architecture cible

Tout le système est orchestré par un **nœud coordinateur** central qui relie les trois
sous-systèmes via des **topics**, **services** et **actions** (à titre d'exemple) :

```mermaid
flowchart LR
  cam["detector<br/>(caméra + IA, J4)"] -->|"/detections<br/>Detection.msg"| coord["mission_coordinator<br/>(params : zones par classe)"]
  coord -->|"PickRequest.srv"| arm["arm_server<br/>(SO-101 + MoveIt, J3)"]
  coord -->|"action NavigateToPose"| base["base<br/>(LeKiwi + Nav2, J2)"]
  arm -->|success| coord
  base -->|result| coord
```

<Aside type="tip" title="Vous avez déjà conçu ce graphe">
Au **Jour 1**, vous avez **conçu et monté un graphe factice équivalent** (détections en dur,
pick simulé, navigation bidon). Le projet final consiste à remplacer chaque nœud factice par
le vrai sous-système des Jours 2 à 4 — **sans changer vos interfaces**.
</Aside>

## Les interfaces à respecter

Pour que les sous-systèmes s'assemblent sans réécriture, **figez vos interfaces** dès
maintenant : une fois définies, on n'y touche plus. Voici une **forme possible** (la vôtre,
conçue au Jour 1, peut différer — l'essentiel est qu'elle reste stable ; un exemple complet
est donné dans [le contrat des briques](https://github.com/EtienneSchmitz/ros2_course/blob/main/plan/briques-j1-pour-j5.md)) :

| Interface | Forme | Producteur → consommateur |
| --- | --- | --- |
| topic de détections (`Detection.msg`) | `string class_id` + `geometry_msgs/Pose pose` | détection (J4) → coordinateur |
| service de saisie (`PickRequest.srv`) | requête `Pose target` → réponse `bool success` | coordinateur → bras (J3) |
| action de navigation (`NavigateToPose`) | goal `PoseStamped` | coordinateur → base (J2) |

## Démarrer

<Steps>

1. **Repartez du graphe factice que vous avez conçu au Jour 1** : vos interfaces, votre nœud
   coordinateur paramétré et votre launch file.

2. **Remplacez les nœuds factices, un par un**, par les vrais sous-systèmes :
   - le nœud de détection factice → la perception du Jour 4 ;
   - le nœud factice du bras → le pick MoveIt 2 du Jour 3 ;
   - le nœud factice de navigation → Nav2 sur le LeKiwi (Jour 2).

3. **Validez le pipeline complet**, puis soignez la robustesse (reprises, cas d'échec).

</Steps>

<Aside type="caution">
Travaillez **par interface**, pas par robot : tant que chaque sous-système publie/consomme
les bons messages, vous pouvez les brancher dans n'importe quel ordre et tester isolément.
</Aside>
