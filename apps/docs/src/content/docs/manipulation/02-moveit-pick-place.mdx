Dans la partie précédente, vous pilotiez le bras **joint par joint**. Ici, vous laissez **MoveIt 2**
faire le travail : vous lui donnez une **pose cible** et il calcule une trajectoire
articulaire sans collision. On construit **brique par brique** (pose nommée → cible
cartésienne → ligne droite → collision), puis on assemble le tout en un **pick & place
complet** que vous codez vous-même.

<Aside type="note" title="Objectifs">
- Comprendre ce qu'apporte MoveIt 2 (IK, planners, planning scene).
- Lancer `demo.launch.py`, planifier et exécuter une trajectoire depuis RViz.
- Maîtriser les briques en code (`moveit_py`) : pose nommée, cible cartésienne, collision.
- Assembler une séquence **pick & place** complète et la tester en simulation.
</Aside>

[MoveIt 2](https://moveit.picknik.ai/) est **la** stack ROS 2 de
planification de mouvement. Au lieu de calculer vous-même chaque
trajectoire articulaire, vous lui donnez un objectif (pose cartésienne,
configuration cible) et il vous renvoie une trajectoire collision-free.

## 1. Ce que MoveIt apporte

- **Cinématique inverse** (IK) : « quelle config de joints donne cette
  pose de la pince ? » — résolu par KDL ou TracIK.
- **Planification** (OMPL, STOMP, Pilz…) : « comment passer de A à B sans
  rentrer dans un obstacle ? ».
- **Planning Scene** : représentation du monde (obstacles, objets attachés).
- **Trajectory execution** : envoie la trajectoire planifiée du bras au
  `arm_controller` (le `joint_trajectory_controller` vu dans la partie précédente) ; la pince est
  pilotée séparément par le `gripper_controller`
  (action `ParallelGripperCommand`).

`move_group` est le node central : il consomme la description du robot
(URDF **+ SRDF**) et expose les services / actions de planification.

## 3. Ce que le SRDF ajoute

L'URDF décrit la **géométrie** ; le **SRDF** (Semantic Robot Description Format)
ajoute la **sémantique** dont MoveIt a besoin — groupes de joints, poses nommées,
paires de collisions à ignorer, end-effector. Pour le SO-101, **il est déjà fourni**
dans le package `so101_moveit_config` de
[so_arm101_ros2](https://github.com/EtienneSchmitz/so_arm101_ros2) : rien à générer,
vous l'utilisez tel quel. Les éléments qui nous serviront :

| Groupe    | Poses nommées disponibles            |
| --------- | ------------------------------------ |
| `arm`     | `home`, `ready`                      |
| `gripper` | `gripper_open`, `gripper_close`      |

La chaîne du bras va de `base_link` (base) à `gripper_link` (effecteur) — ce sont
les deux repères qu'on manipulera en code.

## 4. Premier lancement

```bash
ros2 launch so101_moveit_config demo.launch.py
```

Cela démarre `move_group` + RViz avec le **MotionPlanning panel**
préchargé. Dans RViz :

<Steps>

1. Onglet **Planning** → faites glisser la sphère interactive autour de la pince
   pour définir une cible cartésienne (ou choisissez une **pose nommée** dans le
   déroulant *Goal State*).
2. Bouton **Plan** : MoveIt cherche une trajectoire.
3. Bouton **Execute** : la trajectoire est jouée sur le bras (sim ou réel selon
   le launch).

</Steps>

<Aside type="tip">
Si vous lancez aussi `so101_bringup gazebo.launch.py` (partie Découverte), l'exécution se
répercute dans Gazebo : MoveIt planifie, le `arm_controller` exécute, le bras
bouge dans le simulateur physique.
</Aside>

<Aside type="caution" title="Locale fr_FR">
MoveIt 2 (Kilted) a une régression de locale : en `fr_FR`, le séparateur décimal
« , » casse le parsing de l'URDF. `demo.launch.py` force déjà
`LC_NUMERIC=C.UTF-8` pour contourner. Si vous lancez `move_group` à la main,
exportez la variable d'abord : `export LC_NUMERIC=C.UTF-8`.
</Aside>

## 5. Piloter le bras en code — les briques

RViz, c'est bien pour explorer. Mais un pick & place doit s'enchaîner **tout seul** :
on passe donc au code Python (`moveit_py`). Cette section construit les **quatre
briques** qu'on assemblera ensuite. Chaque extrait s'ajoute au fichier
`pick_and_place.py`.

### 5.1 Se connecter à MoveIt

On instancie `MoveItPy`, on récupère les deux **planning components** (`arm` et
`gripper`), et on se donne deux helpers réutilisables : un pour planifier+exécuter,
un pour fabriquer une `PoseStamped`.

```python
import time

import rclpy
from rclpy.logging import get_logger

from geometry_msgs.msg import PoseStamped
from moveit.planning import MoveItPy

BASE_FRAME = "base_link"
EE_LINK = "gripper_link"


def make_pose(x, y, z):
    """PoseStamped dans base_link, orientation neutre (pince vers le bas simplifiée)."""
    pose = PoseStamped()
    pose.header.frame_id = BASE_FRAME
    pose.pose.position.x = float(x)
    pose.pose.position.y = float(y)
    pose.pose.position.z = float(z)
    pose.pose.orientation.w = 1.0
    return pose


def plan_and_execute(robot, component, logger, sleep_time=0.5):
    """Planifie depuis l'état courant puis exécute. Retourne True si OK."""
    plan_result = component.plan()
    if plan_result:
        robot.execute(plan_result.trajectory, controllers=[])
    else:
        logger.error("Échec de planification.")
    time.sleep(sleep_time)
    return bool(plan_result)
```

### 5.2 Brique 1 — Goal = une pose nommée

Le moyen le plus simple de bouger : viser une **pose nommée** du SRDF. C'est ainsi
qu'on rejoint `home` / `ready` (bras) et qu'on ouvre / ferme la pince.

```python
def move_arm_to_named(moveit, arm, name, logger):
    """name ∈ {'home', 'ready'}."""
    arm.set_start_state_to_current_state()
    arm.set_goal_state(configuration_name=name)
    return plan_and_execute(moveit, arm, logger)


def set_gripper(moveit, gripper, name, logger):
    """name ∈ {'gripper_open', 'gripper_close'}."""
    gripper.set_start_state_to_current_state()
    gripper.set_goal_state(configuration_name=name)
    return plan_and_execute(moveit, gripper, logger)
```

### 5.3 Brique 2 — Goal = une pose cartésienne connue

Pour viser un point précis de l'espace (l'objet, la zone de dépôt), on donne une
`PoseStamped` et le **lien** à y amener (`gripper_link`). MoveIt résout l'IK.

```python
def move_arm_to_pose(moveit, arm, pose, logger):
    """Amène le repère EE_LINK sur `pose`."""
    arm.set_start_state_to_current_state()
    arm.set_goal_state(pose_stamped_msg=pose, pose_link=EE_LINK)
    return plan_and_execute(moveit, arm, logger)
```

<Aside type="tip" title="Volume atteignable du SO-101">
Le bras a un rayon d'action d'environ **0.31 m**. Restez dans cette enveloppe pour
que l'IK trouve une solution :

- **X** (avant) : 0.15 – 0.25 m
- **Y** (latéral) : −0.20 – 0.20 m
- **Z** (hauteur) : 0.05 – 0.15 m

Des valeurs sûres pour tester : objet à `(0.20, 0.0, 0.05)`, dépôt à `(0.0, 0.20, 0.05)`.
</Aside>

### 5.4 Brique 3 — Ligne droite cartésienne (approche / descente)

Avec `set_goal_state`, MoveIt choisit librement le chemin articulaire. Pour une
**descente bien verticale** au-dessus de l'objet, on préfère parfois une **ligne
droite** dans l'espace cartésien : c'est le rôle de `compute_cartesian_path`, qui
interpole entre des **waypoints**.

```python
# Optionnel — descente strictement verticale via une trajectoire cartésienne.
# (Le pick & place du §6 fonctionne aussi avec de simples poses successives.)
waypoints = [make_pose(0.20, 0.0, 0.10).pose, make_pose(0.20, 0.0, 0.05).pose]
# trajectory, fraction = arm.compute_cartesian_path(waypoints, 0.01, 0.0)
```

<Aside type="note">
Pour rester simple, le pick & place ci-dessous **enchaîne des poses cartésiennes**
(au-dessus de l'objet, puis sur l'objet) plutôt que `compute_cartesian_path`. La
version « ligne droite » est un raffinement (voir Exercice 2).
</Aside>

### 5.5 Brique 4 — Sécuriser avec une collision

On déclare la **table** dans le Planning Scene : MoveIt l'évitera automatiquement.
On publie un `CollisionObject` sur `/collision_object`.

```python
from moveit_msgs.msg import CollisionObject
from shape_msgs.msg import SolidPrimitive
from geometry_msgs.msg import Pose

obj = CollisionObject()
obj.id = "table"
obj.header.frame_id = "base_link"

box = SolidPrimitive()
box.type = SolidPrimitive.BOX
box.dimensions = [1.0, 1.0, 0.05]

pose = Pose()
pose.position.x = 0.3
pose.position.z = -0.025

obj.primitives.append(box)
obj.primitive_poses.append(pose)
obj.operation = CollisionObject.ADD
```

## 6. Pick & place complet

On assemble les briques. La séquence enchaîne :
**home → ouvrir pince → approche (+5 cm) → descente → fermer → retrait (+10 cm) →
dépôt → ouvrir pince → home**.

Prérequis, dans **deux terminaux séparés** :

```bash
# Terminal 1 — simulateur
ros2 launch so101_bringup gazebo.launch.py

# Terminal 2 — MoveIt
ros2 launch so101_moveit_config move_group.launch.py
```

Les positions clés, en clair :

```python
OBJECT_XYZ = (0.20, 0.0, 0.05)   # objet à saisir
PLACE_XYZ = (0.0, 0.20, 0.05)    # zone de dépôt
APPROACH_HEIGHT = 0.05           # on s'arrête 5 cm au-dessus avant de descendre
RETREAT_HEIGHT = 0.10            # on remonte de 10 cm après la préhension
```

La fonction `main()` qui orchestre tout :

<Steps>

1. **Démarrage** : connexion à MoveItPy, récupération des groupes `arm` / `gripper`,
   retour à `home`.
2. **Approche** : on ouvre la pince (`gripper_open`), puis on amène `gripper_link`
   **5 cm au-dessus** de l'objet.
3. **Descente** : on rejoint la pose de l'objet.
4. **Préhension** : on ferme la pince (`gripper_close`).
5. **Retrait** : on remonte de 10 cm.
6. **Dépôt** : on rejoint la zone de dépôt (approche puis pose), puis on ouvre la pince.
7. **Retour** : on revient à `home`.

</Steps>

```python
def main():
    rclpy.init()
    logger = get_logger("pick_and_place")

    moveit = MoveItPy(node_name="pick_and_place")
    arm = moveit.get_planning_component("arm")
    gripper = moveit.get_planning_component("gripper")
    logger.info("MoveItPy prêt.")

    ox, oy, oz = OBJECT_XYZ

    # 0. Position de départ
    move_arm_to_named(moveit, arm, "home", logger)

    # 1. APPROCHE — au-dessus de l'objet, pince ouverte
    set_gripper(moveit, gripper, "gripper_open", logger)
    move_arm_to_pose(moveit, arm, make_pose(ox, oy, oz + APPROACH_HEIGHT), logger)

    # 2. DESCENTE — jusqu'à l'objet
    move_arm_to_pose(moveit, arm, make_pose(ox, oy, oz), logger)

    # 3. PRÉHENSION
    set_gripper(moveit, gripper, "gripper_close", logger)

    # 4. RETRAIT
    move_arm_to_pose(moveit, arm, make_pose(ox, oy, oz + RETREAT_HEIGHT), logger)

    # 5. DÉPÔT
    px, py, pz = PLACE_XYZ
    move_arm_to_pose(moveit, arm, make_pose(px, py, pz + APPROACH_HEIGHT), logger)
    move_arm_to_pose(moveit, arm, make_pose(px, py, pz), logger)
    set_gripper(moveit, gripper, "gripper_open", logger)

    # Retour home
    move_arm_to_named(moveit, arm, "home", logger)

    logger.info("Pick & place terminé.")
    moveit.shutdown()
    rclpy.shutdown()


if __name__ == "__main__":
    main()
```

Lancez votre node (le `MoveItConfigsBuilder` doit charger `so101_moveit_config` ;
pensez à `export LC_NUMERIC=C.UTF-8`).

<Aside type="tip">
Aucun objet réel n'est manipulé ici (pas de perception, pas de feedback de
préhension). C'est un **séquencement de trajectoires**. La détection de l'objet
sera traitée au [Jour 4 (Vision & IA)](/vision/) ; l'intégration complète au
[Jours 5-6](/integration/).
</Aside>

## Exercices

Pour aller plus loin une fois le pick & place fonctionnel :

### Exercice 1 — Pose nommée vers pose nommée

Enchaîner deux poses prédéfinies du SRDF (`home` → `ready` → `home`) avec
`move_arm_to_named` (cf. [§5.2](#52-brique-1--goal--une-pose-nommée)).

### Exercice 2 — Descente cartésienne en ligne droite

Remplacez la descente (étape 2 du §6) par une **vraie ligne droite** via
`compute_cartesian_path` (cf. [§5.4](#54-brique-3--ligne-droite-cartésienne-approche--descente)).
Comparez la trajectoire obtenue avec la version « pose à pose ».

### Exercice 3 — Évitement d'obstacle

Ajoutez la **table** (ou une boîte intermédiaire) au Planning Scene
(cf. [§5.5](#55-brique-4--sécuriser-avec-une-collision)) entre départ et cible, puis
vérifiez que MoveIt la contourne. Comparez deux planifications, avec et sans obstacle.

### Exercice 4 — Pick & place avec dépôt contraint

Reprenez la séquence du §6 et imposez une **zone de dépôt** : l'objet doit être
relâché dans le **carré vert de 10 cm** matérialisé dans le monde `pick_place.sdf`.

<Steps>

1. Lancez le monde dédié :
   ```bash
   ros2 launch so101_bringup gazebo.launch.py world:=pick_place.sdf
   ```
2. Adaptez `PLACE_XYZ` pour viser le centre de la zone, `(0.0, 0.20, 0.05)`.
3. Après dépôt, **vérifiez la pose finale** de `gripper_link` :
   ```bash
   ros2 run tf2_ros tf2_echo base_link gripper_link
   ```
   Elle doit tomber dans le carré : `|x| ≤ 0.05` et `|y − 0.20| ≤ 0.05`.
4. **Bonus** : ajoutez le `CollisionObject` « table » (§5.5) et observez l'évitement.

</Steps>
