Avant de passer aux journées du cours, on s'assure que l'environnement est sain. Ces vérifications valent pour les **deux parcours** : installation native ([Ubuntu / WSL](/installation/01-ubuntu/)) ou [Docker](/installation/02-docker/). Dans le cas Docker, exécutez-les **à l'intérieur du conteneur**.

## `ros2 doctor`

```bash
ros2 doctor
```

Doit afficher `All 5 checks passed` (le nombre exact peut varier). Les warnings sur le réseau (`hostname resolves loopback`) sont sans conséquence pour la simulation.

## Le workspace build

```bash
cd ~/ros2_bootcamp_ws
colcon build --symlink-install
```

Le build doit se terminer **sans erreur** (`Summary: N packages finished`).

## Gazebo Ionic s'ouvre

```bash
gz sim shapes.sdf
```

Une fenêtre Gazebo s'ouvre avec quelques formes. Vérifiez que vous pouvez orbiter la caméra à la souris (clic-droit pour panner, molette pour zoomer).

## Le bridge `ros_gz` répond

Dans **trois terminaux** :

<Tabs>
<TabItem label="Terminal 1 — Gazebo">

```bash
gz sim -r shapes.sdf
```

</TabItem>
<TabItem label="Terminal 2 — bridge">

</TabItem>
<TabItem label="Terminal 3 — echo /clock">

```bash

```

</TabItem>

Le terminal 3 doit afficher l'horloge Gazebo qui avance. C'est la preuve que ROS 2 ↔ Gazebo communique.

## Foxglove se connecte au pont

[Foxglove](https://foxglove.dev) est notre outil d'**inspection / monitoring** (complément de RViz, qui reste l'outil d'interaction) : visu 3D web, lecture des valeurs de topics, courbes. Vérifions que l'application se connecte bien au pont.

Lancez le pont :

```bash
ros2 launch foxglove_bridge foxglove_bridge_launch.xml   # ws://localhost:8765
```

Dans l'application Foxglove (installée à l'[étape précédente](/installation/01-ubuntu/), ou via [app.foxglove.dev](https://app.foxglove.dev) dans le navigateur), choisissez **Open connection → Foxglove WebSocket** et saisissez `ws://localhost:8765`.

La connexion doit s'établir et la **liste des topics** apparaître (sélecteur de panneau non vide). C'est la preuve que Foxglove dialogue avec ROS 2.

## Et après ?

À ce stade, la machine est prête, avec les **repos robots** (`lekiwi_ros2`, `so_arm101_ros2`) déjà dans le workspace.

Si `bootcamp.repos` est mis à jour pendant le cours, resynchronisez le workspace avec :

```bash
cd ~/ros2_bootcamp_ws
wget -O bootcamp.repos https://ros2.etienne-schmitz.com/bootcamp.repos
vcs import src < bootcamp.repos   # clone les repos nouvellement ajoutés
vcs pull src                       # git pull main sur les repos déjà clonés
rosdep update
rosdep install --from-paths src --ignore-src -r -y
colcon build --symlink-install
```

`vcs import` clone les **nouveaux** dépôts et les place sur la bonne branche, mais ne récupère
**pas** les nouveaux commits d'un repo déjà présent : c'est `vcs pull` qui fait le `git pull`
sur `main` dans chaque dépôt.
