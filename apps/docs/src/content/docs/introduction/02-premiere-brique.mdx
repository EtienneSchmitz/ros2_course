Place à **votre** projet. Vous allez créer un package ROS 2 et y monter le **graphe du
projet final** (Jour 5) — mais avec des **nœuds factices**. Chaque brique réutilise un
concept de la partie précédente, et sera remplacée par un vrai robot aux Jours 2 à 4.

<Aside type="note" title="Le graphe à construire">
```
[detector] --/detections (Detection.msg)--> [mission_coordinator] --PickRequest.srv--> [arm_server]
                                                    |
                                                    +-- (plus tard) action NavigateToPose --> [base]
```
Aujourd'hui, le `detector` invente des détections, le `arm_server` répond toujours
« pris ». Aux Jours 2-4, vous brancherez la vraie caméra, le vrai bras, le vrai robot
mobile — **sans changer les interfaces**.
</Aside>

## 1. Créer les packages

On sépare les **interfaces** (messages/services) des **nœuds** — c'est la convention ROS 2.

<Steps>

1. **Le package d'interfaces** (`ament_cmake`, car les `.msg`/`.srv` se génèrent en C++) :

   ```bash
   cd ~/ros2_bootcamp_ws/src
   ros2 pkg create mission_interfaces --build-type ament_cmake \
     --dependencies geometry_msgs
   ```

2. **Le package des nœuds** (`ament_python`) :

   ```bash
   ros2 pkg create mission --build-type ament_python \
     --dependencies rclpy geometry_msgs mission_interfaces
   ```

</Steps>

## 2. Brique 1 — le message `Detection.msg`

Créez `mission_interfaces/msg/Detection.msg` :

```text
string class_id
geometry_msgs/Pose pose
```

Déclarez-le dans `mission_interfaces/CMakeLists.txt` :

```cmake
find_package(rosidl_default_generators REQUIRED)
find_package(geometry_msgs REQUIRED)

rosidl_generate_interfaces(${PROJECT_NAME}
  "msg/Detection.msg"
  "srv/PickRequest.srv"
  DEPENDENCIES geometry_msgs
)
```

(et `<member_of_group>rosidl_interface_packages</member_of_group>` dans `package.xml`).

## 3. Brique 1 (suite) — le nœud `detector` (publisher)

Un nœud qui **publie** des détections factices sur `/detections` (concept : **topic**).
Créez `mission/mission/detector.py` :

```python
import itertools
import rclpy
from rclpy.node import Node
from mission_interfaces.msg import Detection


class Detector(Node):
    def __init__(self):
        super().__init__("detector")
        self.pub = self.create_publisher(Detection, "/detections", 10)
        self.classes = itertools.cycle(["rouge", "bleu"])   # détections en dur
        self.create_timer(2.0, self.tick)

    def tick(self):
        msg = Detection()
        msg.class_id = next(self.classes)
        msg.pose.position.x = 0.30
        msg.pose.orientation.w = 1.0
        self.pub.publish(msg)
        self.get_logger().info(f"Détecté : {msg.class_id}")


def main():
    rclpy.init()
    rclpy.spin(Detector())
    rclpy.shutdown()
```

## 4. Brique 2 — le service `PickRequest.srv` + `arm_server`

Créez `mission_interfaces/srv/PickRequest.srv` (concept : **service**) :

```text
geometry_msgs/Pose target
---
bool success
```

Le serveur factice répond toujours « pris ». Créez `mission/mission/arm_server.py` :

```python
import rclpy
from rclpy.node import Node
from mission_interfaces.srv import PickRequest


class ArmServer(Node):
    def __init__(self):
        super().__init__("arm_server")
        self.srv = self.create_service(PickRequest, "/pick", self.on_pick)

    def on_pick(self, request, response):
        self.get_logger().info("Saisie (factice)…")
        response.success = True          # le vrai pick MoveIt arrive au Jour 3
        return response


def main():
    rclpy.init()
    rclpy.spin(ArmServer())
    rclpy.shutdown()
```

## 5. Brique 4 — le `mission_coordinator` (paramètres + orchestration)

Le coordinateur **souscrit** `/detections`, **appelle** le service de pick, et choisit la
**zone de dépôt** selon la classe — zone lue dans un **paramètre** (concept : **paramètres**).
Créez `mission/mission/mission_coordinator.py` :

```python
import rclpy
from rclpy.node import Node
from mission_interfaces.msg import Detection
from mission_interfaces.srv import PickRequest


class MissionCoordinator(Node):
    def __init__(self):
        super().__init__("mission_coordinator")
        # zones de dépôt par classe, surchargeables par paramètre / YAML
        self.declare_parameter("zone_rouge", [1.0, 0.0])
        self.declare_parameter("zone_bleu", [-1.0, 0.0])
        self.create_subscription(Detection, "/detections", self.on_detection, 10)
        self.pick = self.create_client(PickRequest, "/pick")

    def on_detection(self, msg):
        zone = self.get_parameter(f"zone_{msg.class_id}").value
        self.get_logger().info(f"{msg.class_id} → dépôt en {zone}")
        # 1) appeler /pick  2) (Jour 2) envoyer l'action NavigateToPose vers `zone`


def main():
    rclpy.init()
    rclpy.spin(MissionCoordinator())
    rclpy.shutdown()
```

Déclarez les trois exécutables dans `mission/setup.py` :

```python
entry_points={
    "console_scripts": [
        "detector = mission.detector:main",
        "arm_server = mission.arm_server:main",
        "mission_coordinator = mission.mission_coordinator:main",
    ],
},
```

<Aside type="tip" title="Brique 3 — l'action de navigation">
L'envoi vers la zone se fera via l'**action** `NavigateToPose` (client) — branchée sur le
vrai Nav2 du LeKiwi au Jour 2. Aujourd'hui, contentez-vous de logger la zone cible.
</Aside>

## 6. Brique 5 — `mission.launch.py` + le graphe

Assemblez les trois nœuds (concept : **launch file**) dans `mission/launch/mission.launch.py` :

```python
from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
    return LaunchDescription([
        Node(package="mission", executable="detector"),
        Node(package="mission", executable="arm_server"),
        Node(package="mission", executable="mission_coordinator"),
    ])
```

<Steps>

1. **Compiler** et sourcer :

   ```bash
   cd ~/ros2_bootcamp_ws
   colcon build --packages-select mission_interfaces mission
   source install/setup.bash
   ```

2. **Lancer** le graphe :

   ```bash
   ros2 launch mission mission.launch.py
   ```

3. **Visualiser** le graphe (concept : graphe ROS) :

   ```bash
   rqt_graph
   ```

</Steps>

<Aside type="tip" title="Vérifiez votre compréhension">
1. Pourquoi les interfaces sont-elles dans un package séparé (`mission_interfaces`) ?
2. Pourquoi un **service** pour le pick et une **action** pour la navigation ?
3. Que faut-il refaire après avoir modifié un `.msg` ?

<details class="quiz-answers">
<summary>Afficher les réponses</summary>

1. Les `.msg`/`.srv` se génèrent via `ament_cmake` (code C++/Python auto-généré) ; les
   isoler permet à plusieurs packages de les réutiliser sans dépendre du code des nœuds.
2. Le pick est **court avec une réponse** (réussi ou non) → service ; la navigation est
   **longue, avec feedback et annulable** → action.
3. Re-`colcon build` le package d'interfaces puis re-sourcer le workspace.

</details>
</Aside>

3 et 4, vous remplacerez chaque nœud factice par le vrai robot, et au Jour 5 vous
assemblerez le tout.
